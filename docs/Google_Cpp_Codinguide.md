# Codex CLI용 C++ 스타일 적용 지침서
> Google C++ Style Guide를 기준으로 현재 프로젝트에 맞게 Codex CLI가 일관된 코드 수정을 수행하도록 하기 위한 운영 문서

---

## 1. 목적

이 문서는 Codex CLI가 이 C++ 프로젝트를 수정하거나 새 코드를 생성할 때,  
Google C++ Style Guide에 맞는 방식으로 판단하고 적용하도록 유도하기 위한 기준 문서다.

이 문서의 목표는 다음과 같다.

- 코드 스타일의 일관성 유지
- 리뷰 비용 감소
- 자동 수정 시 불필요한 구조 변경 방지
- 기존 코드베이스와 자연스럽게 어울리는 변경 유도
- 가독성과 유지보수성 향상

---

## 2. Codex 작업 원칙

Codex는 항상 다음 원칙을 따른다.

1. **스타일보다 안정성을 우선한다.**
   - 동작을 바꾸지 않아도 되는 경우 로직 변경을 하지 않는다.
   - 스타일 수정은 기능 변경과 분리한다.

2. **작게 수정한다.**
   - 한 번에 너무 많은 파일을 건드리지 않는다.
   - 요청 범위를 벗어난 대규모 리팩터링은 하지 않는다.

3. **기존 구조를 존중한다.**
   - 이미 프로젝트에서 일관되게 사용 중인 패턴이 있다면, Google 스타일과 충돌하지 않는 한 그 패턴을 우선 존중한다.
   - 새 코드는 본 문서 기준으로 작성한다.

4. **설명 가능한 변경만 수행한다.**
   - 각 수정은 “왜 이 방식이 더 읽기 쉽고 안전한지” 설명 가능해야 한다.

5. **스타일 일괄 수정은 별도 작업으로 분리한다.**
   - 기능 구현 PR과 스타일 대청소 PR을 섞지 않는다.

---

## 3. 적용 범위

이 문서는 다음 파일에 적용한다.

- `*.h`
- `*.hpp`
- `*.hh`
- `*.c`
- `*.cc`
- `*.cpp`
- `*.cxx`

다음은 별도 규칙이 우선될 수 있다.

- 외부 라이브러리 코드
- 서드파티 벤더 디렉터리
- 자동 생성 코드
- SQLite amalgamation 같은 외부 배포 원본 파일
- 플랫폼/라이브러리 요구사항 때문에 스타일 예외가 필요한 파일

## 4. 핵심 스타일 규칙

### 4.1 헤더 파일

#### 4.1.1 헤더 가드 사용
모든 헤더 파일은 헤더 가드를 사용한다.

예시:
```cpp
#ifndef COINBOT_ENGINE_MARKET_ENGINE_H_
#define COINBOT_ENGINE_MARKET_ENGINE_H_

// ...

#endif  // COINBOT_ENGINE_MARKET_ENGINE_H_
```

규칙:
- 심볼명은 **프로젝트명 + 경로 + 파일명 + `_H_`** 형태를 기본으로 한다.
- 경로 구분자는 `_`로 변환하고 모두 대문자로 작성한다.
- `#pragma once`만 단독으로 쓰는 방식은 지양하고, 프로젝트 기본은 헤더 가드로 통일한다.

---

### 4.2 Include 규칙

#### 4.2.1 필요한 헤더만 포함
- 사용하는 선언/타입에 필요한 헤더만 include 한다.
- 간접 include에 기대지 않는다.
- “헤더에서 include한 것을 cpp가 그냥 기대는 방식”은 피한다.

#### 4.2.2 include 순서
가능하면 다음 순서를 따른다.

1. 같은 파일의 대응 헤더
2. C 시스템 헤더 / C++ 표준 라이브러리 헤더
3. 서드파티 라이브러리
4. 프로젝트 내부 헤더

예시:
```cpp
#include "engine/market_engine.h"

#include <algorithm>
#include <memory>
#include <string>
#include <utility>
#include <vector>

#include <boost/asio.hpp>

#include "core/logger.h"
#include "exchange/upbit/upbit_client.h"
```

#### 4.2.3 그룹 사이 빈 줄
- include 그룹 사이에는 한 줄 빈 줄을 둔다.
- 같은 그룹 내부에서는 정렬을 유지한다.

---

### 4.3 네이밍 규칙

프로젝트 기본 네이밍은 Google C++ Style Guide 계열에 맞춘다.

- **타입명 / 클래스명 / 구조체명 / enum 타입명**: `PascalCase`
- **함수명**: `PascalCase` 또는 기존 프로젝트 규칙을 따르되, 한 프로젝트 안에서는 통일
- **변수명**: `snake_case`
- **멤버 변수명**: `snake_case_`
- **상수명**: `kPascalCase`
- **매크로**: 전체 대문자 + 언더스코어

예시:
```cpp
class MarketEngine {
 public:
  void Start();
  void Stop();

 private:
  int retry_count_ = 0;
  bool is_running_ = false;
};

constexpr int kDefaultRetryLimit = 3;
```

#### 프로젝트 호환 규칙
현재 코드베이스에 이미 다음 규칙이 널리 퍼져 있다면 강제로 뒤엎지 않는다.

- 기존 public 함수가 `camelCase`
- DTO 필드명이 API 응답 원문과 맞춰져 있음
- DB 스키마/JSON 키와 동일한 이름을 유지해야 함

이 경우:
- **기존 인터페이스는 유지**
- **새 내부 구현 코드에서만 일관성 강화**
- 외부 API/직렬화 포맷과 직접 연결된 이름은 함부로 바꾸지 않는다

---

### 4.4 함수 작성 규칙

#### 4.4.1 함수는 짧고 명확하게
- 한 함수가 여러 책임을 동시에 가지지 않게 한다.
- 이름만 보고 역할이 드러나야 한다.
- 조건 분기와 예외 처리가 길어지면 보조 함수로 분리한다.

#### 4.4.2 매개변수
- 읽기 전용 대형 객체는 `const T&`
- 소형 값 타입은 값 전달
- 소유권 이전이 명확하면 값 전달 후 `std::move`
- nullable 의도가 명확할 때만 포인터 사용

예시:
```cpp
void ProcessOrder(const Order& order);
void SetName(std::string name);
void AttachSession(Session* session);  // null 가능성이 의미를 가질 때만
```

#### 4.4.3 반환값
- 실패 가능성이 분명하면 상태를 표현할 수 있는 반환 구조를 사용한다.
- 출력 인자(out parameter) 남용을 피한다.
- 여러 값을 반환할 때는 의미 있는 struct를 우선 고려한다.

---

### 4.5 클래스 설계 규칙

#### 4.5.1 public이 먼저
클래스 선언 순서는 기본적으로 다음을 따른다.

1. public
2. protected
3. private

#### 4.5.2 생성자와 소멸자
- 단일 인자 생성자는 의도가 불분명하면 `explicit`
- 자원 관리 객체는 RAII를 우선 사용
- 소멸자에서 실패 가능 작업을 하지 않는다

#### 4.5.3 복사/이동 의미를 명확히
- 복사 금지 타입이면 명시적으로 삭제
- 이동 허용 여부를 드러낸다

예시:
```cpp
class WebSocketClient {
 public:
  WebSocketClient();
  WebSocketClient(const WebSocketClient&) = delete;
  WebSocketClient& operator=(const WebSocketClient&) = delete;
  WebSocketClient(WebSocketClient&&) = delete;
  WebSocketClient& operator=(WebSocketClient&&) = delete;
  ~WebSocketClient();

 private:
  // ...
};
```

---

### 4.6 const, constexpr, auto 사용

#### 4.6.1 const 적극 사용
- 변경하지 않는 값/참조/멤버 함수에는 `const`를 명확히 붙인다.
- “나중에 쓸 수도 있으니 일단 mutable” 같은 습관을 피한다.

#### 4.6.2 constexpr 우선
컴파일 타임 상수는 `constexpr`를 우선 사용한다.

```cpp
constexpr int kMaxReconnectAttempts = 5;
```

#### 4.6.3 auto는 의미가 분명할 때만
다음 상황에서 `auto` 사용을 허용한다.

- iterator 타입이 너무 장황할 때
- lambda 반환/복잡한 템플릿 타입
- 우변을 보면 타입이 매우 명확할 때

다음은 지양한다.

```cpp
auto x = GetSomething();  // 타입이 문맥상 불명확하면 지양
```

---

### 4.7 포인터와 참조

- 소유권 없는 필수 대상: 참조
- 소유권 없는 선택 대상: 포인터
- 소유권 있는 동적 객체: `std::unique_ptr` 우선
- 공유 소유권이 정말 필요할 때만 `std::shared_ptr`

원시 포인터는 “소유하지 않음” 의미가 분명할 때 사용한다.

---

### 4.8 제어문과 중괄호

- `if`, `for`, `while` 본문은 한 줄이어도 중괄호 사용을 기본으로 한다.
- 중첩이 깊어지면 조기 반환으로 단순화한다.

예시:
```cpp
if (!is_connected_) {
  return false;
}
```

---

### 4.9 주석 규칙

#### 4.9.1 주석은 “무엇”보다 “왜”
나쁜 예:
```cpp
// i를 1 증가시킨다.
++i;
```

좋은 예:
```cpp
// 재연결 폭주를 막기 위해 최소 백오프를 둔다.
++retry_count_;
```

#### 4.9.2 인터페이스 주석
외부에서 호출되는 핵심 함수는 다음을 설명한다.

- 역할
- 입력값 의미
- 부작용
- 스레드 안전성
- 실패 조건

예시:
```cpp
// 주문 체결 이벤트를 반영해 계정 상태를 동기화한다.
// thread-safe.
// 중복 이벤트가 들어와도 누적 정합성이 깨지지 않도록 설계되어야 한다.
void OnMyTrade(const MyTrade& trade);
```

#### 4.9.3 오래된 주석 금지
코드와 맞지 않는 주석은 없는 것보다 나쁘다.
Codex는 코드 변경 시 관련 주석도 함께 갱신한다.

#### 4.9.4 주석 추가
현재 학습중인 사람의 입장에서 코드가	왜 그렇게 작성되었는지 설명하는 주석을 원칙에 맞게 간단하게 추가한다.

---

### 4.10 에러 처리

- 무시 가능한 에러와 치명적 에러를 구분한다.
- 실패를 숨기지 않는다.
- 로그는 원인 파악에 충분한 문맥을 남긴다.
- assert는 내부 불변식 검사용으로만 사용하고, 런타임 입력 검증을 대체하지 않는다.

---

### 4.11 매크로 사용 최소화

- 상수는 `constexpr`
- 타입 별칭은 `using`
- 짧은 헬퍼는 inline 함수/템플릿 우선
- 매크로는 헤더 보호, 플랫폼 분기, 정말 필요한 경우만 사용

---

### 4.12 전역 상태 최소화

- 전역 변수 사용을 지양한다.
- 싱글턴도 남용하지 않는다.
- 꼭 필요한 공유 상태는 수명, 소유권, 동기화 전략이 드러나야 한다.

---

### 4.13 스레드/동시성 코드 규칙

현재 프로젝트가 멀티스레드/이벤트 기반이라면 특히 다음을 지킨다.

- mutex의 소유 범위를 최소화한다.
- lock을 잡은 상태에서 외부 호출을 피한다.
- 공유 상태는 이름만 보고도 보호 주체가 드러나게 한다.
- 경쟁 조건 회피보다 먼저 **소유권 모델**을 명확히 한다.
- thread-safe 여부를 주석 또는 인터페이스에서 드러낸다.

예시:
```cpp
mutable std::mutex orders_mu_;
std::unordered_map<std::string, Order> orders_;
```

## 5. 포맷팅 규칙

### 5.1 들여쓰기
- 공백 2칸 또는 4칸 중 프로젝트 기존 규칙을 우선한다.
- 이미 프로젝트가 한 가지로 정착돼 있다면 Codex는 그 규칙을 유지한다.
- 새 파일 생성 시에는 프로젝트 다수 파일 기준을 따른다.

### 5.2 한 줄 길이
- 과도하게 긴 한 줄을 피한다.
- 파라미터가 길면 줄바꿈한다.
- 줄바꿈은 읽는 흐름이 자연스러운 위치에서 수행한다.

### 5.3 빈 줄
- 논리적 문단 사이에는 빈 줄을 둔다.
- 너무 촘촘하거나 너무 듬성한 배치를 피한다.

## 6. C++ 프로젝트 특화 예외 규칙

다음 상황은 스타일보다 **실용성/호환성**을 우선할 수 있다.

### 6.1 외부 API DTO
업비트/REST/WS 응답 필드와 직접 매핑되는 DTO는
JSON 키 이름과 최대한 일치시키는 편이 유지보수에 유리할 수 있다.

예:
```cpp
struct MyOrderDto {
  std::string uuid;
  std::string state;
  std::string executed_volume;
};
```

이 경우:
- DTO 필드는 외부 포맷과 정합성을 우선
- 내부 도메인 객체에서는 프로젝트 네이밍 규칙을 적용 가능

### 6.2 DB/직렬화/프로토콜 호환성
다음은 함부로 이름을 바꾸지 않는다.

- DB 컬럼명과 직결된 필드
- JSON 직렬화 키
- 프로토콜 메시지 필드명
- 공개 API 함수명

### 6.3 성능 민감 코드
핫패스에서는 지나친 추상화보다 명확하고 비용이 예측 가능한 코드를 우선한다.
다만 “최적화처럼 보이는 난해한 코드”는 금지한다.

## 7. Codex가 절대 하지 말아야 할 것

Codex는 다음 행동을 하지 않는다.

- 요청하지 않은 대규모 rename
- 공개 인터페이스 이름 일괄 변경
- 파일 전체 포맷 재정렬만 수행하는 과도한 수정
- include 순서만 바꾸고 끝내는 소음성 PR 생성
- 외부 API/JSON/DB 호환성을 깨는 필드명 변경
- thread-safe 보장이 없는 상태에서 동시성 구조 변경
- 테스트 없이 핵심 엔진 로직 대량 수정

## 8. Codex 작업 절차

Codex는 C++ 코드 수정 시 다음 절차를 따른다.

### Step 1. 기존 규칙 파악
- 현재 디렉터리에서 대표 파일들을 읽어 기존 네이밍/들여쓰기/헤더 스타일을 파악한다.
- Google 스타일과 충돌하지 않는 한 기존 일관성을 우선 존중한다.

### Step 2. 변경 범위 최소화
- 요청 파일과 직접 관련된 파일만 수정한다.
- 불필요한 전역 리팩터링은 하지 않는다.

### Step 3. 스타일 반영
- 새 코드/수정 코드에 본 문서 규칙을 반영한다.
- 특히 헤더 가드, include 정리, const 적절성, 이름 일관성, 주석 품질을 확인한다.

### Step 4. 빌드/테스트 우선
- 가능하면 관련 테스트 또는 빌드를 먼저 확인한다.
- 스타일 수정 때문에 컴파일이 깨지지 않도록 한다.

### Step 5. 변경 요약 작성
최종 결과에는 다음을 포함한다.

- 무엇을 바꿨는지
- 왜 바꿨는지
- 기능 변경 여부

## 9. Codex 프롬프트 템플릿

아래 프롬프트를 Codex CLI에서 재사용한다.

### 9.1 스타일 준수 수정 요청
```text
이 저장소의 C++ 코드를 수정할 때 Google C++ Style Guide를 기본 기준으로 따라줘.
다만 기존 프로젝트의 공개 인터페이스와 외부 API/DB/직렬화 호환성은 깨지 말고,
이미 일관되게 사용 중인 네이밍/들여쓰기 규칙은 존중해줘.
변경 범위는 최소화하고, 기능 변경과 스타일 변경을 섞지 말아줘.
수정 후에는 무엇을 왜 바꿨는지 간단히 요약해줘.
```

### 9.2 코드 리뷰 요청
```text
현재 C++ 변경사항을 Google C++ Style Guide 관점에서 리뷰해줘.
다음 항목을 중점적으로 봐줘:
1) 헤더 가드 / include 의존성
2) 네이밍 일관성
3) const / reference / pointer 사용 적절성
4) 클래스 책임 분리
5) 주석 품질
6) 불필요한 전역 상태 / 동시성 위험
문제점과 권장 수정안을 파일 단위로 정리해줘.
```

### 9.3 최소 수정 패치 요청
```text
현재 프로젝트 스타일을 유지하면서 Google C++ Style Guide에 더 가깝게 맞춰줘.
단, 동작 변경 없이 최소 수정만 수행하고,
외부 API와 직렬화 포맷에 영향을 주는 이름 변경은 하지 말아줘.
필요한 파일만 수정해줘.
```

## 10. 리뷰 체크리스트

PR 또는 Codex 결과물 검토 시 아래를 확인한다.

- [ ] 헤더 가드가 올바른가?
- [ ] include가 직접 의존성을 충족하는가?
- [ ] include 그룹/순서가 일관적인가?
- [ ] 클래스/함수/변수 이름이 일관적인가?
- [ ] 멤버 변수 suffix 규칙이 맞는가?
- [ ] `const`를 붙일 수 있는 곳에 붙였는가?
- [ ] 포인터/참조 사용 이유가 명확한가?
- [ ] 함수가 너무 길거나 여러 책임을 가지지 않는가?
- [ ] 주석이 왜 필요한지 설명하는가?
- [ ] 외부 API/DB/직렬화 호환성이 유지되는가?
- [ ] 멀티스레드 코드에서 공유 상태 보호가 분명한가?
- [ ] 스타일 수정이 기능 변경을 유발하지 않았는가?
